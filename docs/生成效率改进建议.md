# 生成效率改进建议

> 注：本文保留为历史候选优化清单。当前执行顺序与完成状态以 `docs/RealismPatchGenerator_性能优化计划表.md` 和 `docs/RealismPatchGenerator_性能优化执行记录.md` 为准，文中旧代码行号可能已经失效。

> 基于对 `RealismPatchGenerator.Core` 生成流水线的深度分析，按影响程度排序。

---

## 🔴 高影响 — 预期可获显著加速

### 1. 文件级并行化

- **位置**: `RealismPatchGenerator.cs:85-88`
- **现状**: 不同输入文件之间完全独立（各自输出到不同的 `sourceFile` key），但当前 `foreach` 顺序处理。
- **改进**: 改用 `Parallel.ForEach` 处理不同文件，各任务持有本地 `generatedItemInfoById`，完成后合并到全局 buffer。多核 CPU 上可接近线性加速。
- **预期收益**: 多核 CPU 上 **2-8x 加速**

### 2. 消除大量冗余 `DeepClone()`

- **位置**: `TemplateCatalog.cs:43-44`、`RealismPatchGenerator.cs:505,587,601,766...`
- **现状**: `JsonNode.DeepClone()` 递归复制整个 JSON 子树，是全项目最大的内存分配源和 GC 压力源。一个模板补丁在生成流程中被 DeepClone 多次（模板加载时双份、生成时反复克隆）。
- **改进**:
  - 模板加载时消除 `templateById` 与 `templates[fileName]` 的双份存储（两者内容完全相同）
  - 对只读数据使用直接引用而非克隆，仅在需要写入时才 Copy-on-Write
- **预期收益**: 内存减半 + GC 压力大幅下降

### 3. 用 `JsonDocument` 替代 `JsonObject` 处理只读数据

- **位置**: `TemplateCatalog.cs:29`、`RealismPatchGenerator.cs:407`
- **现状**: `JsonNode.Parse()` 将整个文件解析为可变树模型，每个节点都有包装开销。
- **改进**: 对于只读的模板、规则文件，使用 `JsonDocument`（基于 UTF8 buffer 的只读解析），只在需要修改的场景才转换为 `JsonObject`。
- **预期收益**: 内存和解析时间降低 30-50%

---

## 🟡 中影响

### 4. 模板全量加载改懒加载

- **位置**: `TemplateCatalog.cs:9-71`
- **现状**: `Load()` 一次性把所有模板目录下所有 JSON 文件读入内存并 DeepClone。实际对某个输入文件，往往只需要其中几个模板文件。
- **改进**: 改为按需加载，未被使用的模板文件不读入内存。
- **预期收益**: 启动和冷加载显著加速

### 5. 预缓存模板类型与允许字段集合

- **位置**: `RealismPatchGenerator.cs:2136-2162`、`RealismPatchGenerator.cs:2615-2624`
- **现状**: `GetAllowedOutputFieldMap` 每次遍历模板文件中所有模板条目的全部字段来构建允许字段集合；`IsWeapon()` 每次遍历整个模板文件查找 `$type` 字段。
- **改进**: 在模板加载时预计算这些结果并缓存（如 `Dictionary<string, bool> weaponTypeCache`、`Dictionary<string, HashSet<string>> allowedFieldCache`）。
- **预期收益**: 避免 O(n) 遍历 → O(1) 查找

### 6. Profile 推断使用 Aho-Corasick 批量匹配

- **位置**: `ProfileInferenceService.cs:370-581`、`RealismPatchGenerator.cs:3282-3284`
- **现状**: `ContainsAnyKeyword` 中 `keywords.Any(k => text.Contains(k))` 是 O(n*m) 的字符串匹配，每个物品的 profile 推断会执行数十次此类调用。
- **改进**: 对不变的关键词列表预构建 Aho-Corasick 自动机，或使用 .NET 8+ `SearchValues` 进行高效多模式匹配。
- **预期收益**: Profile 推断加速，减少字符串比较开销

### 7. `StaticData.CreateDefaultXxxTemplate()` 改为单例

- **位置**: `StaticData.cs:238-394`
- **现状**: 这些方法每次调用都新建含 30+ 字段的 `JsonObject`，在 `CreateAllowedPatchFieldSet` 等路径中频繁调用。
- **改进**: 改为 `static readonly` 单例，遍历时直接使用不复制。
- **预期收益**: 减少不必要的堆分配

## 🟢 低影响

### 8. 模板 I/O 并行化

- **位置**: `TemplateCatalog.cs:28`
- **现状**: 所有模板文件用 `File.ReadAllText` 顺序读取。
- **改进**: 改用 `Parallel.ForEach` 或 `Task.WhenAll` 并行读取。
- **预期收益**: 模板加载时间降低

### 9. 输出写入并行化

- **位置**: `PatchOutputPipeline.cs:25`
- **现状**: 不同输出文件完全独立，但在一个 `foreach` 中顺序写入。
- **改进**: 使用 `Parallel.ForEach` 并行写入不同输出文件。
- **预期收益**: 输出阶段加速

### 10. 日志列表预分配容量

- **位置**: `RealismPatchGenerator.cs:34`
- **现状**: 处理数百物品时 `List<string> logs` 多次扩容。
- **改进**: 预分配 `new List<string>(estimatedCapacity)`。
- **预期收益**: 避免扩容开销
