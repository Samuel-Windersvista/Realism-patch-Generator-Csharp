# 性能文档改写与清理设计稿

## 背景

当前 `docs` 目录中的性能相关文档存在三个问题：

1. `生成效率改进建议.md` 中部分建议方向正确，但有些结论写得过满，未充分体现代码语义边界、并发安全前提和可变对象风险。
2. `RealismPatchGenerator_性能优化计划表.md` 混合了“规划态”和“已完成态”内容，容易让读者误判哪些工作仍待执行、哪些只是历史记录。
3. `RealismPatchGenerator_第一阶段拆分落地顺序表.md` 与 `RealismPatchGenerator_第二阶段拆分落地顺序表.md` 已属于阶段性过程文档，继续保留在当前主文档集合中会干扰后续性能工作判断。

本次工作目标不是重构整个 `docs` 目录，而是对性能相关文档做一次中等强度清理：保留当前有效指导文档，保留有历史价值的执行记录，移除已完成使命的阶段性路线文档。

---

## 目标

建立一组职责清晰、互不冲突的性能文档：

1. **当前建议**：明确哪些优化建议成立、哪些仅部分成立、哪些存在风险或需缩小适用范围。
2. **当前计划**：给出按风险/收益/前置条件排序的可执行落地计划表。
3. **历史记录**：保留已有性能优化执行结果，避免历史数据丢失。
4. **清理过时文档**：删除已完成阶段的拆分路线文档，避免其继续被当作当前行动依据。

---

## 范围

### 本次会处理的文件

- 改写：`docs/生成效率改进建议.md`
- 改写：`docs/RealismPatchGenerator_性能优化计划表.md`
- 视情况补充说明：`docs/RealismPatchGenerator_性能优化执行记录.md`
- 删除：`docs/RealismPatchGenerator_第一阶段拆分落地顺序表.md`
- 删除：`docs/RealismPatchGenerator_第二阶段拆分落地顺序表.md`

### 本次不会处理的内容

- 不修改任何生产代码或测试代码
- 不新增性能测试实现或 profiling 工具
- 不重整整个 `docs` 目录的命名体系
- 不处理与性能主题无直接关系的规则说明、使用说明、文章草稿等文档

## 设计方案

### 1. `docs/生成效率改进建议.md`

该文档将从“直接列出 10 条优化建议”改为“技术评估文档”，核心职责是给出经过代码核查后的判断。

建议结构：

1. 背景与评估范围
2. 总体结论
3. 逐项建议评估
   - 成立
   - 部分成立
   - 不建议按原表述直接执行
4. 调整后的优先级排序
5. 建议先做 / 后做 / 暂不做
6. 与现有历史文档的关系说明

重点要写清楚的技术边界：

- 文件级并行化需要先解决 `patchOutputBuffer`、`generatedItemInfoById`、`logs` 等共享状态的线程安全问题。
- `DeepClone()` 的冗余与否必须分路径讨论：模板加载阶段存在明显冗余；生成链路中的克隆不应一概视为可删除。
- `JsonDocument` 更适合只读模板/规则路径，不应不分场景替换所有 `JsonObject` 解析入口。
- `StaticData.CreateDefaultXxxTemplate()` 若直接改单例，会引入共享可变对象污染风险。

### 2. `docs/RealismPatchGenerator_性能优化计划表.md`

该文档将从“混合规划+完成状态记录”改为“当前仍可执行的性能落地计划”。

建议结构：

1. 目标与适用范围
2. 当前代码现实约束
3. 优先级分层
   - P0：低风险高收益
   - P1：中风险中高收益
   - P2：高复杂度，需要进一步 profile 证明再做
4. 可执行任务表
   - 任务目标
   - 涉及文件
   - 实施内容
   - 验证方式
   - 风险点
   - 完成标准
5. 建议执行顺序（按轮次）
6. 暂不建议投入的事项

计划内容应避免把“已经完成过的一轮优化”再写成待办，而应聚焦：

- 仍值得推进的模板元数据缓存
- 关键词匹配与 profile 推断热点优化
- 安全的输出侧并发点
- 有前置条件的文件级并发
- 仅在只读路径考虑的 JSON 表示优化

### 3. `docs/RealismPatchGenerator_性能优化执行记录.md`

该文档保留为历史记录，不再作为当前行动计划使用。

处理策略：

- 尽量不改动其已有数据与结论
- 如有必要，只在开头增加一句说明，明确“当前建议与计划以新版文档为准”

### 4. 过时文档清理

删除以下文件：

- `docs/RealismPatchGenerator_第一阶段拆分落地顺序表.md`
- `docs/RealismPatchGenerator_第二阶段拆分落地顺序表.md`

删除依据：

- 它们描述的是阶段性拆分路径，且文档内部已写到首轮落地完成态。
- 当前主线任务已转向“性能建议校准与落地计划”，继续保留会让读者把历史拆分路线误读为当前待执行事项。
- 相关历史事实若仍有价值，应由 `性能优化执行记录.md` 和新版计划文档承接，而不是继续保留旧阶段路线图。

## 方案取舍

本次采用的是“**当前版本文档集 + 清理旧阶段文档**”方案，而不是：

- 只原地修修补补、不处理过时文档
- 或对整个 `docs` 目录做全面重编目

选择该方案的原因：

1. 能满足“更准确建议 + 可执行计划 + 中等清理”的目标。
2. 保留必要历史记录，但让当前文档职责更明确。
3. 改动范围可控，不会把任务扩大成全面文档重构工程。

---

## 验收标准

本次文档改写与清理完成后，应满足：

1. 性能相关主文档只保留三类角色：当前建议、当前计划、历史记录。
2. `生成效率改进建议.md` 能明确区分“成立 / 部分成立 / 有风险”的建议。
3. `性能优化计划表.md` 不再混入“已全部完成”的执行记录口吻，而是清晰表达剩余可做事项。
4. 已完成使命的第一阶段、第二阶段拆分路线文档被移除。
5. 各文档之间表述不互相冲突。

---

## 风险与控制

### 风险 1：误删仍有参考价值的文档

控制方式：仅删除已完成的阶段性路线文档；保留执行记录文档作为历史证据。

### 风险 2：新计划与历史记录冲突

控制方式：明确区分“当前建议/计划”和“历史执行记录”的文档角色，在执行记录开头补充短说明。

### 风险 3：计划写得过满，变成不现实的全量优化清单

控制方式：按风险/收益重新排序，只保留当前仍有价值且具备验证路径的任务。
