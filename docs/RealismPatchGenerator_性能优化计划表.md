# RealismPatchGenerator 性能优化计划表

## 目标

这份计划表用于约束“拆分完成后，真正进入性能优化时准备怎么做”。

当前判断是：

- 第一阶段和第二阶段的拆分方案已经足以构成性能优化前提条件
- 原因不是“已经优化了”，而是“优化落点已经被切清楚了”
- 因此接下来的性能工作应建立在明确模块边界、明确指标、明确回归验证之上，而不是继续在主类里盲目改热点

---

## 性能优化的前提条件

真正开始性能优化前，应至少满足以下条件：

1. 模板加载、输入路由、ItemInfo 构造、输出落盘有独立边界
2. 规则执行入口明确，不再由主类同时承担推断、范围修正、价格计算与分发
3. profile 推断与分析上下文有可缓存的独立落点
4. 每类热点都能被局部验证，而不是必须依赖整条 Generate 链路做黑盒回归

按你当前已经确定的两阶段方案，这四条是可以满足的。

---

## 优化目标

性能优化的目标不是单纯追求一次极限提速，而是按收益和风险分层推进：

- 先降低重复计算
- 再降低重复分配
- 再降低重复扫描
- 最后再考虑 I/O 写出方式与更重的结构优化

这意味着性能优化顺序应优先围绕：

- 上下文缓存
- 推断缓存
- alias / 模板索引
- DeepClone 与重复 JSON 处理
- 输出阶段的重复拼装与写出

---

## 基线与验收口径

开始每一轮优化前，先固定一组统一的性能观察口径：

### 观测指标

- 总生成耗时
- 模板加载耗时
- 单文件处理耗时
- 规则执行耗时
- 输出写出耗时
- 总分配量或 GC 压力变化

### 推荐基线场景

1. 小样本：单个 user_templates 文件
2. 中样本：一个子目录
3. 大样本：整个 input 目录完整生成

### 每次优化后的验收要求

- 生成结果不变
- 现有测试继续通过
- 至少一个基线场景性能有可测改进
- 若性能提升不明显但复杂度明显增加，则不继续扩大这类优化

---

## 建议执行顺序

建议按四轮推进，而不是一次性并行开工。

1. 第 0 轮：性能基线与计时点
2. 第 1 轮：分析上下文与推断缓存
3. 第 2 轮：模板/别名/规则热点索引化
4. 第 3 轮：DeepClone 与 JSON 处理减量
5. 第 4 轮：输出流水线与文件写出优化

---

## 第 0 轮：性能基线与计时点

### 目标

先拿到可信的“优化前数据”，避免后面只能靠体感判断快慢。

### 计划内容

- 给 Generate 主流程补最小计时点
- 记录模板加载、文件处理、规则执行、输出写出四段耗时
- 选定固定输入样本，形成后续对比基线

### 建议落点

- RealismPatchGenerator 主流程编排层
- 未来的 PatchRuleApplier / PatchOutputPipeline

### 预期收益

- 后续每轮优化都能量化收益
- 能更容易识别真正热点，而不是凭感觉优化

### 风险控制

- 计时只做轻量统计，不引入复杂 profiling 基础设施

---

## 第 1 轮：分析上下文与推断缓存

### 目标

优先消灭重复字符串处理、重复 token 切分、重复 profile 推断。

### 计划内容

- 为单个 item 构造一次 PatchAnalysisContext 并在生成链内复用
- 为 ProfileInferenceService 增加单项级别缓存
- 合并重复的 GetLowerText、tokenize、armor/caliber/variant 文本拼接

### 重点热点

- InferWeaponProfile
- InferGearProfile
- InferModProfile
- InferAmmoProfile / InferAmmoSpecialProfile / InferAmmoPenetrationTier

### 预期收益

- 这通常是投入最小、收益最稳定的一轮
- 对中大样本批量生成的收益会比较明显

### 风险控制

- 缓存键必须只基于稳定输入，不允许依赖可变 patch 中间状态
- 先做生成链内缓存，不要上来就做全局缓存

---

## 第 2 轮：模板/别名/规则热点索引化

### 目标

减少线性扫描与重复匹配，尤其是 clone alias 和模板关联查找。

### 计划内容

- 为 alias 匹配补预处理索引或预分词结果
- 复核 templateById / templateFileByItemId / templateParentIndex 的查询路径
- 对高频规则关键字匹配建立更适合当前数据结构的查找缓存

### 重点热点

- TryResolveTemplateAlias
- 模板文件回推与 parent/template 解析
- attachment/gear 名称兜底判断中的高频关键词匹配

### 预期收益

- 降低某些输入格式下 clone / alias 解析的放大成本
- 减少随着模板总量增长带来的扫描退化

### 风险控制

- 先做只读索引，不改变匹配优先级
- 若命中顺序本身带语义，索引必须保留原优先级结果

---

## 第 3 轮：DeepClone 与 JSON 处理减量

### 目标

减少 JsonNode / JsonObject 的重复克隆、重复解析与重复字段扫描。

### 计划内容

- 审计模板加载阶段的双重 DeepClone 是否可以减到一次
- 审计 patch 构造与 finalize 阶段的重复复制
- 复核是否存在同一 patch 多次遍历全字段的情况

### 重点热点

- 模板加载
- patch merge / finalize
- 输出前字段修正与全局夹紧

### 预期收益

- 降低分配量和 GC 压力
- 对完整目录生成更容易体现收益

### 风险控制

- 这一轮最容易引入共享引用污染，必须坚持最小改动与回归验证
- 凡是有“源模板不可被污染”约束的路径，不能为了少 clone 直接复用可变对象

---

## 第 4 轮：输出流水线与文件写出优化

### 目标

最后处理输出阶段的重复组织和落盘成本。

### 计划内容

- 评估 PatchOutputPipeline 内是否存在重复分组、重复排序、重复序列化
- 视收益决定是否改成更流式的 JSON 写出
- 评估是否要减少中间态字符串构造

### 重点热点

- SavePatches
- 输出分组整理
- JSON 序列化写盘

### 预期收益

- 对完整 output 场景能进一步减少尾部耗时
- 更适合在前几轮 CPU 热点下降后再做

### 风险控制

- 不要过早引入复杂的流式写出实现
- 如果输出阶段只占总耗时很小比例，就不应投入过多复杂度

---

## 不建议优先做的性能优化

以下内容在前四轮之前不建议优先处理：

- 并行化整个生成流程
- 一上来更换 JSON 技术栈
- 先做大规模对象池化
- 先把所有 helper 改成 Span 或极端微优化写法

原因是这些方案虽然可能有收益，但会显著提高复杂度和验证成本，且在当前阶段不一定命中真正热点。

---

## 每轮建议产出

每轮性能优化结束后，建议固定留下三类产出：

1. 更新 CHANGELOG 中的性能条目
2. 记录优化前后耗时对比
3. 记录是否有结果一致性或测试覆盖补充

这样后面即使继续迭代，也能知道每一轮到底带来了什么收益。

---

## 备注

这份计划表的核心原则是：

- 先拆清边界
- 再测清热点
- 再做最值得的优化

也就是说，性能优化应是“沿着拆分后的模块边界逐轮推进”，而不是重新回到一个大而全的主类里做试错式修改。
